CPM Benchmark
Report Agent System

A fully autonomous media intelligence pipeline that collects real CPM pricing data, stores it in a cloud database, synthesizes AI insights, and emails a daily report — whether or not your Mac is on.

The architecture that was built:
A Vercel serverless cron runs at 8am EST daily, pulling CPM web data via Brave Search, reading/writing 210+ rows in Neon Postgres, and sending HTML email reports via Resend. Mac-side scripts serve as a local backup and dev environment.

System Architecture

Every service this system touches, and how they connect. Vercel is the spine.

Two runners, one pipeline. Vercel is primary and runs in the cloud regardless of whether your Mac is on. The Mac scripts (launchd) are a backup that runs the same logic locally using DuckDB instead of Neon. Both write to GitHub for the monthly CSV commit; only Vercel writes to Neon Postgres.
Section 3

The Agents — What Each One Does

Each script has a single, well-defined job. Here's the full picture.

⚡ cpm-report.js — Vercel Cron Handler (Primary)

Lives in cpm-vercel/api/cpm-report.js. This is the production runner — it fires via Vercel's built-in cron scheduler at 8am EST every day, whether or not any Mac is running. It's a Node.js serverless function that has 10 seconds to complete and uses environment variables set in the Vercel dashboard (never committed to git).

What it does, in order:

  • Authenticates via Authorization: Bearer CRON_SECRET header (blocks unauthorized triggers)
  • Calls Brave Search API for real-time CPM pricing data across ad channels
  • Reads existing CPM rows from Neon Postgres cpm_benchmarks table for historical context
  • Builds AI synthesis by combining historical data + fresh web findings into an HTML email report
  • Sends the formatted HTML email via Resend REST API to EMAIL_TO
  • Writes new verified CPM data points to Neon Postgres (deduplicated via UNIQUE(year, month, channel))
  • Logs run outcome, data points found, and timestamp to the agent_runs table
  • On the 1st of each month: encodes the monthly CSV and commits it to GitHub via Contents API
  • Returns {"ok":true,"outcome":"autonomous","dataPoints":N,"historicalRows":210}

💻 daily-report.mjs — Mac Backup Runner

Lives in agent/daily-report.mjs (repo) and ~/Documents/cpm-agent/agent/cpm-report-agent/daily-report.mjs (working copy). Runs via macOS launchd as a backup at 8am. Uses the same pipeline logic but reads from a local cpm_benchmarks.parquet file via DuckDB instead of Neon — no network database connection required.

Key differences from the Vercel version:

  • Reads CPM data from cpm_benchmarks.parquet via DuckDB (in-process, zero latency)
  • Uses FORCE_UPDATE=true env flag to bypass the once-per-month data guard
  • Credentials come from a local .env file (never in git)
  • Triggered by com.rod.cpm-report.plist at ~/Library/LaunchAgents/
  • Logs stdout/stderr to ~/Documents/cpm-agent/logs/
  • After any git changes, copy the updated script: cp agent/daily-report.mjs ~/Documents/cpm-agent/agent/cpm-report-agent/

🔍 quality-agent.mjs — System Validator

Lives in agent/quality-agent.mjs. Run this any time you want to confirm the Mac-side system is healthy, especially after setting up a new Mac or rotating credentials. It runs 9 check sections and emails a color-coded HTML report.

What it validates:

  • Env vars: All required keys present (BRAVE_API_KEY, GITHUB_TOKEN, RESEND_API_KEY, EMAIL_TO)
  • Files: Scripts, parquet file, plist, and node_modules all exist
  • DuckDB: Can open and query parquet — confirms 210 rows readable
  • Brave API: Live HTTP call returns valid response
  • Resend: API key format valid (re_ prefix check)
  • GitHub: Token valid and repo accessible
  • launchd: Schedule loaded with correct Node binary path
  • Script integrity: Key code patterns present in daily-report.mjs
  • Auto-fix: --fix flag detects and corrects wrong Node path in plist

🖥️ setup-new-mac.sh — One-Command Mac Setup

Lives in agent/setup-new-mac.sh. Designed to be run on a brand new Mac with a single curl | bash command. Takes roughly 5 minutes and leaves the Mac fully configured and scheduled.

What it installs and configures:

  • Xcode Command Line Tools, Homebrew, nvm, Node.js v22
  • Clones github.com/rodbrathwaite79/cpm-malloy-model
  • Installs DuckDB with the correct binary for your chip (ARM64 or x86)
  • Copies scripts to working directory at ~/Documents/cpm-agent/agent/cpm-report-agent/
  • Generates a machine-specific com.rod.cpm-report.plist with the correct Node path
  • Prompts for all required credentials and writes .env (skips if already present)
  • Loads the launchd schedule with launchctl load
  • Runs quality-agent.mjs as final validation
Section 4

How a Daily Run Works

From Vercel cron trigger to inbox delivery — the Vercel path (primary).

Vercel cron fires at 8:00 AM EST
Vercel's scheduler triggers POST /api/cpm-report with the Authorization: Bearer CRON_SECRET header. This runs in Vercel's cloud — no Mac needed. Simultaneously (or as fallback if Vercel is down), the Mac's launchd plist triggers daily-report.mjs at 8am local time.
Brave Search — fetch real-time CPM web data
The Brave Search API returns up to 10 web results for queries like "CPM advertising benchmark 2026." Each result's title, URL, and description are captured. These feed the AI prompt and appear as clickable source links in the email — grounding every claim in a real, dated web source.
Neon Postgres — read historical CPM rows
The Vercel function connects to Neon via DATABASE_URL and queries the cpm_benchmarks table — 210+ rows of historical CPM data spanning multiple years, broken down by channel (paid social, paid search, programmatic display, video/CTV, streaming audio). This provides the trend context the AI uses for synthesis.
Build AI insights + HTML email
Historical CPM data + Brave web excerpts are combined into an AI synthesis prompt. The model returns 3–4 structured insights with source citations. These insights, along with a horizontal bar chart and 6-month trend table, are assembled into an email-safe HTML document via lib/report-html.js.
Send email via Resend
The HTML is POSTed to Resend's /emails endpoint using the RESEND_API_KEY. Resend is a pure REST API — no SMTP, no OAuth, no app passwords. A 200 response confirms queued delivery. Free tier covers 3,000 emails/month.
Write new CPM rows to Neon Postgres
Any verified new CPM data points from the web search are written back to the cpm_benchmarks table. The schema uses UNIQUE(year, month, channel) so duplicate rows are silently skipped via ON CONFLICT DO NOTHING. This is how the database grows over time.
Log run outcome to agent_runs table
A row is inserted into agent_runs with: run_date, source (vercel or mac), outcome (autonomous / error), data_points_found, and created_at. This is the system's audit trail — visible via GET /api/metrics without any credentials.
1st of month only: commit CSV backup to GitHub
When new CPM data is verified and it's the 1st of the month, the monthly additions are serialized to cpm_monthly_updates.csv and pushed to the GitHub repo via the Contents API (base64-encoded PUT). This keeps a human-readable backup of every new row in git history — redundancy on top of Neon.

New Mac Setup & Troubleshooting

Everything you need to get a fresh Mac running and diagnose common failures.

One-Command Setup

# Run this on the new Mac — takes ~5 minutes
curl -sSL https://raw.githubusercontent.com/rodbrathwaite79/cpm-malloy-model/main/agent/setup-new-mac.sh | bash

# After setup: set credentials
nano ~/Documents/cpm-agent/agent/cpm-report-agent/.env

# Verify everything is healthy
node ~/Documents/cpm-agent/agent/cpm-report-agent/quality-agent.mjs
Apple Silicon note: If you see a DuckDB architecture error after setup, the ARM64 binary wasn't installed correctly. Fix with: cd ~/Documents/cpm-agent/agent/cpm-report-agent && rm -rf node_modules && npm install

Updating Scripts After Git Changes

# Pull latest from GitHub
cd ~/Documents/cpm-agent/malloy-model-git && git pull

# Copy to working directory
cp agent/daily-report.mjs ~/Documents/cpm-agent/agent/cpm-report-agent/
cp agent/quality-agent.mjs ~/Documents/cpm-agent/agent/cpm-report-agent/

# Verify
node ~/Documents/cpm-agent/agent/cpm-report-agent/quality-agent.mjs

Architecture Summary

The original design ran everything on a Mac — powerful, but fragile (Mac must be on, single point of failure, local-only data). The current design flips this: Vercel is the primary runner, Neon Postgres is the database, and Resend handles email — all cloud, all always-on. The Mac scripts remain as a development environment and launchd backup, running identical logic against a local parquet file. The Malloy model (index.malloy) sits alongside the data for semantic querying via the CPM Reports connector. Two runners. One pipeline. Zero single point of failure.
