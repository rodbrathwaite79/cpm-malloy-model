Section 1

System Components

Six components. Three run automatically in the cloud. Three run on your Mac. One semantic model ties the data together.

⚡
Cloud · Primary
Vercel Cron Handler
cpm-vercel/api/cpm-report.js

Fires at 8am EST daily via Vercel's cloud scheduler — no Mac required. Searches Brave for CPM data, reads/writes Neon Postgres, builds an HTML email report, sends via Resend, and logs every run to the agent_runs table.

🤖
Guild · ROI Engine
Guild ROI Agent
agent/agent.ts

The metrics and intelligence layer. Tracks 8 ROI dimensions across all runs, logs every user change request as a HITL event, measures improvement rate over time, and generates AI insights from CPM data. Six input modes — pipeline runs, change requests, implementation confirmations, custom queries, read-only dashboard, and reset.

🗂️
Cloud · Semantic Layer
Malloy Semantic Model
index.malloy

Defines pre-built analytical views over cpm_benchmarks.parquet: channel rankings, YoY comparison, seasonal patterns, blended monthly averages. Served via Malloyyo on Vercel. Queried by the Guild agent via the CPM Reports MCP connector — no SQL required.

💻
Mac · Backup
Daily Report Script
agent/daily-report.mjs

Mac-side backup runner. Triggered by launchd at 8am. Runs the same pipeline logic as the Vercel function but reads a local cpm_benchmarks.parquet file via DuckDB instead of Neon Postgres. Useful for development and offline testing.

🔍
Mac · QA
Quality Agent
agent/quality-agent.mjs

9-section health checker. Validates all env vars, DuckDB reads 210 rows, Brave/Resend/GitHub APIs respond, launchd plist is loaded with correct Node path, and script integrity patterns are present. Emails a color-coded HTML report. Run with --fix to auto-repair the plist.

🖥️
Mac · Setup
Mac Setup Script
agent/setup-new-mac.sh

One curl | bash command sets up the entire Mac-side system from scratch: Homebrew, nvm, Node 22, repo clone, DuckDB (ARM64-aware), launchd plist generation, credential prompting, and a Quality Agent health check at the end.

Section 2

8 ROI Dimensions

Every metric the Guild agent now tracks, why it matters to a customer, and how it's calculated. All persisted in Guild's state store across sessions.

01
Automation Rate
What % of pipeline runs completed with zero human intervention. The headline metric — the higher it is, the more the agent is doing the job alone.
autonomousRuns / totalRuns × 100
↑ Higher = better. Target: >90%.
02
Cost Per Output
Real LLM token cost per report run, pulled from Guild's guild_get_daily_llm_usage() API. Makes the agent's cost of operation concrete and auditable.
totalEstimatedCostUsd / totalRuns
↓ Lower = better. Typically fractions of a cent.
03
Human Time Recovered
Dollar value of analyst time the agent replaced. Based on autonomous runs × estimated 15 min per report × configurable hourly rate. The "before vs. after" number customers can take to a CFO.
autonomousRuns × 15min × $100/hr
↑ Higher = more value delivered.
04
ROI Score
Human time saved divided by total LLM spend. Expresses return as a multiple — "for every $1 spent on LLM calls, the agent returned $X in analyst time." The number that closes deals.
humanTimeSaved / totalLLMCost
Typical result: 500x–2000x. Excellent ≥ 10x.
05
Change Request Rate
How often a human had to ask the agent to change something — format, content, channels, frequency, recipients. Each request is a HITL event logged with type, description, session ID, and date. Declining rate = agent maturing.
totalChangeRequests / totalRuns
↓ Declining trend = agent earning trust.
06
Implementation Rate
Of all change requests logged, what % were marked as implemented. Closing the loop: a request was logged, a change was made, the agent confirmed it. Proves the feedback cycle is working.
implementedChanges / totalChangeRequests
↑ Higher = tighter human–agent feedback loop.
07
Error Rate
Pipeline failures as a % of total runs. Tracked separately from HITL — a run that needed human help is different from a run that crashed. Declining error rate proves reliability is improving over time.
errorRuns / totalRuns × 100
↓ Lower = more reliable. Target: <5%.
08
Data Coverage
Average number of verified CPM data points found per run. Output quality metric — more data points means the agent found more real, sourced, usable market intelligence per report.
sum(dataPointsFound) / totalRuns
↑ Higher = richer intelligence per run.
Why 8 dimensions instead of one ROI number? A single ROI multiple is easy to game and easy to dismiss. Showing how the ROI is composed — automation rate, cost, coverage, change request trend — makes it defensible to a CFO, a skeptical buyer, and a renewal conversation. Each dimension answers a different objection.
Section 3

HITL Tracking — Two Distinct Kinds

The original system tracked only one kind of HITL. The new system tracks two — and they tell completely different stories.

⚠️ Type 1 — Pipeline HITL

The agent tried to do its job autonomously and couldn't. Something in the data pipeline broke down.

1
Vercel cron fires on the 1st of the month, searches Brave for new CPM data
2
No verifiable data found — extraction returns null for all channels
3
Agent sends a HITL email: "Please check adsposure.com / emarketer.com manually"
4
Neon logs outcome: "hitl" — counted against automation rate
Logged in: cpm-report.js → Neon agent_runs table
💬 Type 2 — User Change Request (NEW)

A human proactively asked the agent to change something. The pipeline worked — but the output wasn't right.

1
User comes to Guild and says: "Update the email format to include a bar chart"
2
Agent receives { changeRequest: { type: "format", description: "..." } }
3
Logs it to Guild state: date, session ID, type, description, implemented: false
4
Change is built. Agent called with markImplemented — loop closed, implementation rate updated
Logged in: agent.ts → Guild task.save() state
Why this distinction matters for Guild's ROI story: Type 1 HITL measures reliability — did the data pipeline succeed? Type 2 HITL measures quality and trust — is the agent producing output that humans accept without asking for changes? A declining Type 2 rate over time is the clearest signal that an agent is maturing. It's the metric that answers "is this agent getting better?" — and that's the renewal conversation.

The Improvement Signal

The Guild agent now computes a trend on the change request rate — comparing the last 10 runs vs. the prior 10. This is the "improvement signal" that no other metric captures.

📉
Declining
Change request rate falling over time. The agent is learning what good looks like. Trust is increasing.
➡️
Stable
Consistent rate. The agent is reliable but not improving. May need new change types implemented.
📈
Rising
More requests per run over time. The agent may have regressed, or the user's expectations have grown.
Section 4

Guild Agent — 6 Input Modes

The Guild agent (agent.ts) handles six distinct input types. Each mode updates a different part of the state or triggers a different output.

A
Pipeline
Automated Pipeline Run
Called by: Vercel cron or daily-report.mjs after each report
The standard production run. Receives historical CPM rows + web findings, generates AI insights, records token cost via Guild LLM API, and increments all run counters. Sets outcome to autonomous, hitl, or error.
{ "rows": [...], "webFindings": [...], "outcome": "autonomous", "dataPoints": 5 }
B
HITL Log
User Change Request
Called by: anyone who opens the Guild agent and requests a modification
Logs the interaction as a HITL event. Records the type of change (format, content, channels, frequency, recipients, other), the description, the session ID, and the date. Does not call the LLM — just updates state and confirms the log.
{ "changeRequest": { "type": "format", "description": "Add a bar chart to the email" } }
C
Close Loop
Mark Change Implemented
Called by: developer/operator after shipping a change that was requested
Closes the loop on a change request by session ID. Marks it as implemented: true, records the implementation date and notes, and recalculates the implementation rate. The feedback cycle is now measurable end-to-end.
{ "markImplemented": { "sessionId": "abc123", "notes": "Added Chart.js bar chart to email template" } }
D
Custom Query
Custom CPM Question via Malloy
Called by: analyst or end user with a specific research question
Same as Mode A but the question field overrides the default executive summary prompt. Data is typically supplied by querying the Malloy semantic model through the CPM Reports MCP connector, then passed as rows.
{ "rows": [...], "webFindings": [...], "question": "What is the YoY trend for CTV CPMs?" }
E
Dashboard
Read-Only Metrics Dashboard
Called by: anyone who wants to see current ROI metrics without running a report
Detected when no rows, web findings, data points, or outcome are provided. Shows the full 8-dimension ROI dashboard and change request history without recording a new run. Clearly labeled "read-only — test run" in output.
{ }  // or omit all fields
F
Reset
Reset All State
Called by: operator when starting fresh or after a demo
Wipes all persisted state — run history, change requests, counters, cost totals — back to zero. Irreversible. Confirmed with a clear success message. Does not affect Neon Postgres data (only Guild agent state).
{ "reset": true }
Section 5

Where Malloy Fits In

Malloy is the semantic layer between raw data and the Guild agent. It answers the question: "why can't the agent just query the database directly?"

🗃️
Raw Data
cpm_benchmarks.parquet on GitHub. 210+ rows: year, month, channel, CPM, source. A flat file — no schema, no context, no business logic.
🧩
Malloy Semantic Model
index.malloy — pre-defined analytical views: channel_ranking, yoy_comparison, seasonal_pattern, by_channel_month, blended_monthly. Served via Malloyyo on Vercel.
🤖
Guild Agent
Queries Malloy via the CPM Reports MCP connector. Asks "give me channel_ranking" — gets back structured data. No SQL. No schema knowledge required.
The MCP connector is the bridge. The CPM Reports MCP server (connected in this session as mcp__2ae5c6a3…) exposes query(), list_sources(), and describe_source() tools. When the Guild agent calls query("run: cpm_benchmarks -> channel_ranking"), it gets back the pre-computed ranking without ever touching a database connection string. Malloyyo handles serving the model; the MCP handles the API contract.

Without Malloy vs. With Malloy

Without Malloy
  • Agent receives raw CSV/parquet rows and must figure out grouping, sorting, and aggregation itself
  • Every question requires knowing the schema column names
  • Adding a new view (e.g., seasonal pattern) requires changing the agent code
  • No reusable business logic — each agent reinvents the same queries
With Malloy
  • Agent requests named views: channel_ranking, yoy_comparison, seasonal_pattern
  • Business logic lives in the model, not the agent — one place to update
  • New analytical views are added to index.malloy without touching agent code
  • Any agent or tool that connects via MCP gets the same consistent data contract

The 5 Pre-Built Views

by_channel_month

Full time series — every month × every channel, sorted chronologically. The main view used for the daily report and trend analysis.

channel_ranking

Channels sorted by average CPM descending, with min/max range. Answers "which channel is most expensive to advertise on?"

yoy_comparison

Same month across different years, by channel. Answers "is paid social getting more or less expensive compared to last year?"

seasonal_pattern

Average CPM by calendar month across all years. Answers "which months are cheapest to run campaigns?" (typically Jan–Feb).

annual_by_channel

Annual average, min, and max CPM per channel. Useful for budget planning and year-end benchmarking.

What This System Now Proves

The original question was: "what is the ROI of a Guild agent, from a customer's perspective?" The answer is now measurable across 8 dimensions — not a single number a CFO can wave away, but a complete picture: how often did the agent run without help, what did it cost, what did it replace, how many times did a human have to correct it, is it getting corrected less over time, and is every correction being acted on?

The change request rate and its trend are the most strategically important metrics Guild doesn't have yet anywhere else. A declining rate across a customer's fleet of agents is the dataset that proves agents aren't just automating tasks — they're earning autonomy. That's the story that turns a renewal into an expansion.
