How Everything Fits Together

A plain-English guide to what each piece of the system does, how they connect, and how we measure value.

The Pieces of the System

Eight services work together. Here's what each one does.

Think of it like a restaurant kitchen. Vercel is the stove (does the actual cooking on a timer). Neon is the walk-in fridge (stores everything safely). GitHub is the recipe book (holds all the instructions). Resend is the waiter (delivers the result to you). Brave Search is the food supplier (brings in fresh ingredients weekly).
⚡

Vercel PRIMARY ENGINE

A cloud platform that hosts and runs the CPM report. It has a built-in timer (called a "cron") that fires every Monday at 8am ET — no server, no computer needed.

Like having a self-cleaning oven that turns itself on every Monday. You set it once; it runs forever.
🗄️

Neon Postgres DATABASE

The cloud database that stores everything: 210+ historical CPM data points and every AI interaction logged. Neon is managed, meaning it backs itself up and never goes offline.

Like a shared Google Sheet that never gets accidentally deleted, is always fast, and can be read from anywhere in the world.
🔍

Brave Search API DATA SOURCE

The system searches the web using the Brave Search API (not a browser — a direct programmatic query). It pulls current CPM prices from industry reports and publisher pages.

Like hiring a research assistant to Google "current CPM rates 2026" every Monday and send you a summary.
📧

Resend EMAIL DELIVERY

Resend is the email API that sends the Monday report. It's simpler than Gmail and doesn't need OAuth setup — just an API key. Sends a fully formatted HTML email with charts and tables.

Like a postal service that accepts a pre-written letter and guarantees delivery — no stamps, no trips to the post office.
🐙

GitHub CODE + LOGGING

GitHub stores all the code (the repo is cpm-malloy-model) and also runs the session logging pipeline. When you push a new session log file, a GitHub Action automatically posts it to the database.

Like a filing cabinet that also has a robot inside — when you drop a new document in, the robot reads it and files a copy in the database automatically.
💬

Claude / Cowork AI ASSISTANT

Claude (this AI) is the primary tool for building and maintaining the system. Each Cowork session is automatically logged with a task description, time estimate, and dollar value. Claude also writes its own session log at the end of every session.

Like a contractor who not only does the work, but also automatically fills out the time sheet when they're done.
🖥️

Mac launchd BACKUP RUNNER

A Mac scheduling system that runs the CPM report locally at 8am every Monday as a backup. If Vercel were ever to fail (it hasn't), the Mac would still run the report. Installed on your MacBook.

Like a backup generator — you never need it, but it's there if the main power goes out.
⚙️

Guild AGENT RUNTIME

Guild hosts and runs the cpm-report-agent TypeScript agent — a more advanced version of the CPM report that uses an AI model. Guild handles execution, state management, scheduling via triggers, and full session-level audit logs for every run.

Like a managed server for your AI agent — it runs your code, tracks every run, and shows you exactly what happened and how much it cost.

What Gets Logged Per Session

Every session record contains these fields. Claude fills them in automatically.

The "What"

  • Provider — Claude, ChatGPT, etc.
  • Tool — Cowork, Claude Code, cursor, chat
  • Task type — code, analysis, research, document
  • Description — 1–2 sentences of what was done
  • Output file — what artifact was produced

The "How Much"

  • Human hours — estimated time for a skilled human
  • Dollar value — hours × market rate
  • API cost — actual money spent (if measurable)
  • First pass? — did AI get it right first try?
  • Corrections — how many fixes were needed

Automatic Session Logging — How It Works

Zero manual data entry. Claude tracks itself.

The problem it solves: Previously, if you wanted to log an AI session, you'd have to manually fill out a form after every session. That's like a contractor having to remember to bill you — it never happens consistently. The new system makes Claude responsible for its own time sheet, then a cloud robot files it automatically when you push to GitHub.
1

Session Start (automatic)

When a Cowork session begins, Claude captures a timestamp and creates a session log file in session-logs/pending/. This happens on the very first tool call of the session.

2

During the Session (automatic)

After each major task, Claude updates the log with what was done, the start and end time, the task type, estimated human hours, and dollar value. All fields filled in by Claude — no input from you.

3

Session End — "log this" (one word from you)

When you say "log this" or "wrap up," Claude finalizes the JSON, writes the summary, and commits the file to the git repo. You then push to GitHub with git push.

4

GitHub Action fires (fully automatic)

The moment the push arrives at GitHub, a cloud job (GitHub Action) detects the new file, validates the JSON, and POSTs it to the /api/log-interaction endpoint on Vercel.

5

Database write + archive (fully automatic)

Vercel writes the record to Neon Postgres. The GitHub Action then moves the file from pending/ to posted/ and commits the change with [skip ci] so it doesn't loop.


Why GitHub Actions instead of a local script? A local script only runs on the computer it's installed on. GitHub Actions runs in the cloud — so it works whether you're on your MacBook Pro, a different Mac, or any other machine. You push from anywhere; the logging happens automatically.

Guild Agent — Internals & Governance

How the hosted TypeScript agent works, and how to stay in control of it.

What Guild does: Guild is the platform that hosts and runs the cpm-report-agent TypeScript agent in the cloud. Every time you publish a new version (guild agent save --publish), Guild compiles the code, validates it, and makes it available to run. Every run creates a session with a full audit trail.

How the Babel Compiler Works (and Why It Matters)

The agent uses Guild's AutomaticallyManagedStateAgent pattern. You write a plain async run() function, and Guild's Babel compiler secretly transforms it into a resumable state machine — code that can pause mid-run, save its state, and pick up exactly where it left off after a tool call completes or the user responds.

⚠️ Critical maintenance rule: The Babel compiler cannot handle { ...spreadSyntax } or [...arrays] when the source is an imported variable. Any spread using an import or module-level constant inside the run() body will cause the build to fail with a NotImplemented: SpreadElement error. Always use Object.assign() and .concat() instead.

Pattern to AvoidUse InsteadWhy
{ ...importedVar, key: val }Object.assign({}, importedVar, { key: val })Spread on non-local label fails at build time
[...arr, newItem]arr.concat([newItem])Same SpreadElement restriction

Governance — Staying in Control

Guild gives you three layers of visibility into every agent run.

📊 Usage Insights (Token Cost)

Shows LLM token consumption over time. This is your cost-governance view — if a run consumed 10× the usual tokens, you'll see the spike here.

URL: app.guild.ai → Usage Insights

To cap costs: Settings → LLM Settings → Daily token limit. To use your own Anthropic key (BYOK): Settings → LLM Settings → Add API key.

📋 Session Audit Trail

Every agent run creates a session. Sessions show status (Completed / Failed), every tool call made, and the full event stream — like a flight recorder for each run.

guild session list — all recent runs
guild session events <id> — every step
guild session tasks <id> — tool calls only

🔖 Version History

Every guild agent save creates a version. Published versions have a version number; failed builds stay as DRAFT. The history tells you exactly what code was live at any point in time.

guild agent versions — full list
Currently live: v1.0.12 · PUBLISHED

⏰ Trigger Management

A Guild time trigger fires the agent on schedule (Monday 8am). You can pause, resume, or inspect past runs from the trigger — without touching the agent code.

guild trigger list — all triggers
guild trigger sessions <id> — scheduled run history
guild trigger deactivate <id> — pause without deleting

How Dollar Values Are Calculated

Market rates for skilled contractor work, applied to estimated human hours.

The method: For each task, Claude estimates how many hours a skilled contractor would have needed — not how long Claude took (often seconds), but how long a human would take working alone. That estimate is multiplied by the market rate for that type of work.
Task Type What It Means Hourly Rate Example
Code Writing or modifying software, building features, fixing bugs $150/hr Building the GitHub Actions workflow
Analysis Evaluating options, debugging, diagnosing issues, reviewing systems $175/hr Diagnosing the 401 → 400 API errors
Research Investigating topics, gathering information, comparing options $125/hr Researching cross-machine logging approaches
Document Writing guides, documentation, HTML files, reports $125/hr Writing the Skills Reference HTML guide

Examples from actual sessions:

• Built an interactive 18-skill HTML reference guide → 5 human hours × $150/hr = $750
• Switched CPM report from daily to weekly (5 files) → 0.5 hours × $150/hr = $75
• Designed the cross-machine logging architecture → 2 hours × $175/hr = $350
• Validated end-to-end pipeline via Chrome → 1 hour × $175/hr = $175
